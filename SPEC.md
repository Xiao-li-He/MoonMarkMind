# Markdown 双向转换规范

本文档定义 Markdown 与脑图树之间的双向映射规则，目标是让结构解析、编辑操作和导出行为都可预测、可实现、可维护。

## 1. Markdown -> 脑图

### 1.1 目标

1. 允许用 `#` 到 `######` 表示前 6 层结构。
2. 允许用列表继续表示第 7 层及以后，支持无限层级。
3. 保证标题和列表混用时，层级关系可被唯一解析。
4. 保证实现时可以用栈或树遍历稳定解析。

### 1.2 基本定义

1. 整篇 Markdown 对应一棵树。
2. 树中存在一个隐式根节点 `ROOT`。
3. `ROOT` 不显示在最终脑图中。
4. 每个可见节点包含以下信息：
   - `text`：节点文本
   - `children`：子节点列表
   - `depth`：相对 `ROOT` 的层级深度
   - `type`：`heading` 或 `list_item`

### 1.3 标题规则

1. `#` 表示第 1 层节点。
2. `##` 表示第 2 层节点。
3. `###` 表示第 3 层节点。
4. `####` 表示第 4 层节点。
5. `#####` 表示第 5 层节点。
6. `######` 表示第 6 层节点。
7. 标题的层级由 `#` 的数量决定。
8. 标题节点的父节点必须是最近的、层级更浅的祖先节点。
9. 标题最多只负责前 6 层结构，不再继续增加标题层级。

### 1.4 列表规则

1. 无序列表符号 `-`、`*`、`+` 视为等价。
2. 有序列表 `1.`、`2.`、`3.` 视为等价。
3. 列表项的层级不由序号决定，而由其父节点和缩进结构决定。
4. 列表项的深度恒等于其父节点深度加 1。
5. 列表可以无限嵌套，用于表达第 7 层及以后结构。
6. 列表项可以挂在标题下，也可以挂在另一个列表项下。

### 1.5 层级定义

1. 文档根为 `ROOT(depth=0)`。
2. `#` 节点为 `depth=1`。
3. 每个标题或列表项的子节点，深度都比父节点大 1。
4. 因此：
   - `#` 的直接子节点深度为 2。
   - `######` 的直接子节点深度为 7。
5. “第几层”统一按脑图深度计算，不单独区分 Markdown 语法层和脑图层。

### 1.6 标题与列表的衔接

1. 标题下面可以直接跟列表项。
2. 标题下面的第一个列表项，默认就是该标题的直接子节点。
3. 列表项不需要额外缩进才能作为标题子节点。
4. 缩进的作用是决定列表内部的父子关系，不是决定能否挂到标题下面。

### 1.7 示例

```md
# 根
- 一级分支
  - 二级分支
    - 三级分支
## 另一个主题
- 子分支
```

对应关系：

1. `根` 为第 1 层。
2. `一级分支` 为第 2 层。
3. `二级分支` 为第 3 层。
4. `三级分支` 为第 4 层。
5. `另一个主题` 为第 2 层。
6. `子分支` 为第 3 层。

### 1.8 混合规则

1. 标题优先构成主干结构。
2. 列表用于在标题下继续展开分支。
3. 如果标题和列表混用，解析器应始终按最近有效父节点挂接。
4. 不允许把一个列表项同时挂到多个父节点下。

### 1.9 缩进规则

1. 同层列表项应保持相同缩进。
2. 子列表项的缩进必须严格大于父列表项。
3. 建议统一使用 2 空格或 4 空格缩进。
4. 不建议混用 tab 和空格。
5. 缩进减少时，解析器应回退到最近可匹配的祖先节点。

### 1.10 非结构内容

1. 粗体、斜体、删除线、行内代码、链接等，不生成新层级。
2. 这些内容只作为节点文本或样式保留。
3. 段落文本若未被定义为结构节点，可作为当前节点说明文字处理。
4. 代码块默认不拆分为层级节点。

### 1.11 无限层规则

1. 前 6 层优先使用标题表达。
2. 第 7 层及以后统一使用列表表达。
3. 列表可以继续无限嵌套。
4. 超过 `######` 的结构深度，不再通过标题扩展，而通过列表延伸。

### 1.12 解析优先级

1. 先识别标题，建立主干。
2. 再识别列表，挂接到当前主干或当前分支下。
3. 再处理列表嵌套关系。
4. 最后处理文本格式类内容。

### 1.13 推荐实现策略

1. 使用一个栈维护当前路径。
2. 遇到标题时，弹出所有层级不小于当前标题的节点。
3. 新标题挂到栈顶节点下。
4. 遇到列表项时，挂到当前最近的有效父节点下。
5. 进入更深层列表时，将该列表项压栈。
6. 退出列表层级时，弹栈回到正确父层。

### 1.14 约束

1. 一个节点只能有一个直接父节点。
2. 层级必须单向递增或回退，不允许跳跃式产生歧义。
3. 同级节点必须共享同一父节点。
4. 结构树必须是有穷、无环、单根树。

### 1.15 简版规则

如果只保留最核心的实现约定，可以简化为：

1. `#` 到 `######` 表示第 1 到第 6 层。
2. 列表表示第 7 层及以后。
3. 列表层级由缩进决定。
4. 每个节点的深度都等于父节点深度加 1。
5. `ROOT` 是隐式根，不显示。

### 1.16 节点移动规则

1. `上移` 采用复合语义，优先表示同级兄弟节点之间的向上移动。
2. 当当前节点存在前一个兄弟节点时，`上移` 只改变兄弟顺序，不改变层级。
3. 当当前节点已经是该层级的第一个兄弟节点时，继续 `上移` 会提升一级。
4. 提升一级后，当前节点会从原父节点下移到父节点的父节点下，成为其同级节点。
5. 若提升后的插入位置需要确定，建议固定为“插入到原父节点之前”，以保证行为一致。
6. 当当前节点已经是最顶层节点且位于最前时，再次 `上移` 应视为无操作。
7. `上移` 的默认含义不应同时包含“换序”和“提升层级”两个独立结果，实际行为应由当前所处位置唯一决定。

## 2. 脑图 -> Markdown

### 2.1 目标

1. 脑图中的每个节点都要能序列化回 Markdown。
2. 结构操作只改变树结构，不改变文本内容。
3. 内容操作只改变节点文本或节点附带的正文块，不改变兄弟关系。
4. Markdown 作为导出格式时，应遵循一套固定的“规范化输出”。

### 2.2 数据模型

每个节点至少包含：

1. `text`：节点标题文本
2. `children`：子节点
3. `body`：附加内容块，按顺序保存
4. `level`：逻辑层级
5. `type`：结构类型，通常为 `heading` 或 `list_item`

另有一个隐式根节点：

1. `ROOT` 不导出为 Markdown
2. 所有顶层节点都挂在 `ROOT` 下

### 2.3 层级映射

#### 2.3.1 标题层

1. 第 1 层 -> `#`
2. 第 2 层 -> `##`
3. 第 3 层 -> `###`
4. 第 4 层 -> `####`
5. 第 5 层 -> `#####`
6. 第 6 层 -> `######`

#### 2.3.2 列表层

1. 当层级超过 6 时，改用列表表达
2. 列表项的层级由它的父节点决定：
   - 子节点深度 = 父节点深度 + 1
3. 无序列表符号 `-`、`*`、`+` 等价
4. 有序列表 `1.`、`2.` 等价，序号本身不参与层级计算

#### 2.3.3 统一约定

1. 前 6 层优先用标题
2. 第 7 层及以后用嵌套列表
3. 这是一条规范化导出规则，不是唯一真理，但必须在系统里固定下来

### 2.4 结构操作规则

#### 2.4.1 新增分支

1. 含义：在当前节点的同级位置后面新增一个兄弟节点
2. Markdown 结果：
   - 如果当前节点是标题，则新增同级标题
   - 如果当前节点是列表项，则新增同级列表项

例子：

```md
## 主题 A
## 主题 B
```

或者：

```md
- 分支 A
- 分支 B
```

#### 2.4.2 增加子节点

1. 含义：在当前节点下面新增一个子节点
2. Markdown 结果：
   - 新节点深度 = 当前节点深度 + 1
   - 若新深度 <= 6，用标题表示
   - 若新深度 > 6，用列表表示

例子：

```md
# 根
## 子节点
```

或更深时：

```md
###### 第六层
  - 第七层
```

#### 2.4.3 删除

1. 含义：删除当前节点及其整个子树
2. Markdown 结果：
   - 该节点及其所有后代都不再输出
3. 默认规则：
   - 删除是整棵子树删除
   - 不自动提升子节点，不自动重挂到别的父节点

#### 2.4.4 上移 / 下移

1. 含义：只改变同级兄弟节点的顺序
2. 不改变层级
3. 不改变父节点
4. 不改变子树结构
5. Markdown 输出时，只需要调整同级块的先后顺序

#### 2.4.5 提升 / 降低层级

如果你的编辑器支持这种操作，建议单独定义：

1. 提升一级：当前节点成为原父节点的同级兄弟
2. 降低一级：当前节点成为前一个同级兄弟的子节点

注意：

1. 上移 / 下移 不等于 提升 / 降低
2. 上移 / 下移 是排序
3. 提升 / 降低 是改树结构

### 2.5 内容操作规则

#### 2.5.1 编辑文本

1. 只修改节点的 `text`
2. 不改变层级
3. 不改变兄弟顺序
4. Markdown 中直接改标题行或列表项文本即可

#### 2.5.2 插入链接

推荐两种规则，必须二选一并固定：

1. 规则 A：作为节点文本的一部分
   - 例如 `[链接文字](https://example.com)`
   - 适合短链接或标题内链接
2. 规则 B：作为节点正文块
   - 例如插在节点下面的一段正文里
   - 适合较长说明

如果要做严格可逆，推荐：

1. 链接优先作为 inline 内容
2. 节点正文里也允许出现链接，但应保留原始 Markdown

#### 2.5.3 插入图片

1. 推荐作为节点正文块，不建议默认塞进标题文本里。
2. 导出时可用：

```md
![alt](url)
```

3. 小图标、装饰性图片可以作为 inline 或正文内容
4. 大图、说明图建议作为节点 body

#### 2.5.4 插入代码块

1. 代码块应作为节点正文块，而不是节点标题的一部分
2. 导出时使用 fenced code block：

```md
```ts
const a = 1
```
```

3. 代码块属于当前节点，不属于子节点，除非用户明确创建了子节点

#### 2.5.5 插入列表

这里要分清两种“列表”：

1. 结构列表：表示子节点树
2. 正文列表：表示节点的说明内容

建议规则：

1. 如果用户执行的是“增加子节点 / 新增分支”，那就是结构列表
2. 如果用户执行的是“插入列表”，那就是正文列表

正文列表示例：

```md
## 说明
- 条目 A
- 条目 B
```

这两个 `-` 是正文内容，不是脑图子节点。

### 2.6 结构列表 vs 正文列表 的区分

这是最容易混淆的点，所以建议明确：

#### 2.6.1 结构列表

1. 来自“新增分支 / 增加子节点”
2. 会变成脑图树的 `children`
3. 参与层级计算

#### 2.6.2 正文列表

1. 来自“插入列表”
2. 作为节点内容保留
3. 不参与脑图树结构
4. 只在 Markdown 节点正文里显示

如果不区分这两种，导出会很容易出现“到底这串 bullet 是节点还是说明”的歧义。

### 2.7 Markdown 输出顺序

建议采用以下规范化顺序：

1. 先输出节点标题行
2. 再输出该节点的正文块
3. 最后输出子节点

这样一来：

1. 节点文本先出现
2. 说明内容紧随其后
3. 树结构最后展开

这对阅读和反向解析都更友好。

### 2.8 根节点规则

1. `ROOT` 是隐式根，不导出
2. 所有 `#` 节点都是 `ROOT` 的可见子节点
3. 如果文档中有多个顶层主题，它们都属于同一个隐式根

也就是说，Markdown 文档可以有多个 `#`，它们在脑图里是同层兄弟节点。

### 2.9 规范化导出策略

为了让“脑图 -> Markdown”稳定、可预测，建议固定以下策略：

1. `1~6` 层优先用标题
2. `7+` 层用列表
3. 同级节点按编辑器顺序输出
4. 删除节点时删除整个子树
5. 上移 / 下移只改顺序，不改层级
6. 插入的链接、图片、代码块默认进入节点正文
7. “新增分支 / 增加子节点”才生成结构性 Markdown 层级

### 2.10 示例

脑图结构：

1. 根
   - 分支 A
     - 子节点 A1
       - 子节点 A1-1
   - 分支 B

导出 Markdown：

```md
# 根
## 分支 A
### 子节点 A1
#### 子节点 A1-1
## 分支 B
```

如果深度超过 6：

```md
###### 第六层
- 第七层
  - 第八层
```

### 2.11 推荐实现接口

如果后面要写编辑器，建议把操作接口拆成这些：

1. `addSibling(node)`
2. `addChild(node)`
3. `deleteNode(node)`
4. `editText(node, text)`
5. `moveUp(node)`
6. `moveDown(node)`
7. `indent(node)` 或 `promote(node)`
8. `insertInline(node, content)`
9. `insertBlock(node, block)`

这样导出 Markdown 时就很好做映射。

### 2.12 最终约束

1. 结构操作只改树
2. 内容操作只改文本或正文块
3. 标题只负责前 6 层
4. 列表负责第 7 层及以后
5. 正文列表和结构列表必须分开定义
6. `ROOT` 不出现在最终 Markdown 中
